Scenario Page

Real-world SmartDict cases

These recipes show how SmartDict features combine in practice. Each case is intentionally shaped like a real config someone would maintain.

Scenario 1

ML experiment configuration

python 
smartdict.parse({
    "dataset": "spotify",
    "load": {
        "base_path": "~/data/${dataset}",
        "dev_path": "${load.base_path}/dev",
    },
    "network": {
        "num_hidden_layers": 3,
        "num_attention_heads": 8,
    },
    "store": "checkpoints/${dataset}/${network.num_hidden_layers}L${network.num_attention_heads}H/",
})

This pattern is the bread-and-butter SmartDict use case: nested references remove repeated path construction and reduce config duplication.

Scenario 2

Embedding model fallback chain

smartdict.parse({
    "repr_source_model": "text-embedding-3-small",
    "embedding_model": "${sid_embedding_model:${repr_source_model:null}}",
})
# {'repr_source_model': 'text-embedding-3-small', 'embedding_model': 'text-embedding-3-small'}

This is useful when a more specific field should override a broad default, but the config should still remain typed if everything is missing.

Scenario 3

Typed runtime settings

smartdict.parse({
    "debug": "${env.DEBUG:false|bool}",
    "port": "${env.PORT:8000|int}",
    "save_dir": '${dataset:"My Dataset"|strip|lower|slug}',
})

Pipelines make config values usable immediately, instead of forcing the application to perform ad hoc type conversion everywhere.

Scenario 4

JSON metadata fallback

smartdict.parse({
    "metadata": '${config:{"hello": "world", "owner": "jyonn"}}',
})

Use JSON defaults when you want nested objects or arrays without manually decoding strings after parsing.

Scenario 5

Object-backed profile switching

class Config:
    def __init__(self):
        self.profile = "prod"

smartdict.parse({
    "app": Config(),
    "services": {
        "prod": {"url": "https://example.com"},
        "dev": {"url": "http://localhost:8000"},
    },
    "selected_url": "${services.${app.profile}.url}",
})

This pattern is handy when runtime state is represented by a small object but environment details still live in plain configuration dictionaries.
